输入url到展示的过程

分类： js常见问题 标签： javascript
<p>作为前端工程师，我们日常打交道的就是浏览器了，可是你想过当你输入url回车到页面展示的这一个过程都发生了什么吗？</p> <h3 id="浏览器渲染之前主要有如下几步"><a href="#浏览器渲染之前主要有如下几步" class="headerlink" title="浏览器渲染之前主要有如下几步"></a>浏览器渲染之前主要有如下几步</h3><ul> <li><p>浏览器根据DNS域名服务器得到域名对应的公网ip</p> </li> <li><p>浏览器对这个ip发送http请求</p> </li> <li><p>服务器接受、处理http请求，并响应</p> </li> <li><p>浏览器收到服务器响应的内容并解析</p> </li> <li><p>DNS解析：DNS服务器对url进行一对一的ip映射解析<br>一个域名解析的过程：<a href="http://www.baidu.com/">www.baidu.com</a> -&gt; 网络服务商 -&gt; 递归服务器 -&gt; 根DNS服务器 -&gt; .com顶级DNS服务器 -&gt; baidu.com权威DNS服务器 -&gt; IP返回给用户<br>为了下次查找方便，所以递归服务器就会记录下来 就叫做DNS缓存，这一层也是黑客进行DNS攻击的薄弱地方。</p> </li> <li><p>DNS攻击：因为递归服务器会把解析记录存下来，所以我们只需要让他错一个错误的缓存记录就可以了。域名和DNS服务器采用问答的方式去请求和响应，当询问的端口号和回答的端口号相同时就会把该条记录存下来。最早该端口号是规律递增的，黑客只需要赶在DNS服务器响应前进行答复，所以dns缓存极易被污染。</p> </li> <li><p>TCP三次握手：解析完后，就是服务器和浏览器的交互，交互之前就需要创建连接，我们知道tcp协议是全双工传输，需要浏览器和服务端都可以传输数据，都需要确认对方可以接收到，所以需要三次握手</p> <ul> <li>ACK表示消息收到</li> <li>SYN表示请求创建连接</li> <li>FIN表示请求关闭连接</li> </ul> </li> <li><p>TCP四次挥手：而连接的关闭需要四次，需要双方都确认关闭</p> </li> <li><p>发送请求，分析 url，设置请求报文(头，主体)</p> </li> <li><p>服务器返回请求的文件 (html)</p> </li> <li><p>浏览器进行渲染</p> <ul> <li>HTML parser –&gt; DOM Tree<ul> <li>标记化算法，进行元素状态的标记</li> <li>dom树构建</li> </ul> </li> <li>css parser –&gt; Style Tree<ul> <li>解析 css 代码，生成样式树</li> </ul> </li> <li>集合 –&gt; Render树<ul> <li>结合dom树和样式树，生成渲染树</li> </ul> </li> <li>layout 根据渲染树布局</li> <li>Gup 绘制：根据样式进行页面绘制</li> </ul> </li> <li><p>解析dom遇到js代码的过程：因为dom解析和js执行是两个引擎完成，为防止渲染发生不可预知的错误，js引擎和渲染引擎是互斥的，所以会阻断dom解析</p> <ul> <li>阻断dom解析</li> <li>下载并执行js</li> <li>继续dom解析</li> </ul> </li> <li><p>执行的js操作cssDom的情况：</p> </li> </ul> <ul> <li>阻断dom</li> <li>执行js(操作css)</li> <li>下载css并解析cssDom</li> <li>执行js</li> <li>继续dom解析</li> </ul>
